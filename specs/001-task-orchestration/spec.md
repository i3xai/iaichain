## Clarifications

### Session 2026-06-15

- Q: 子任务/聚合结果的质量验证由什么机制判定通过？ → A: 自动质量门禁——规则校验 + 裁判模型（model-as-judge）评分，达阈值即通过
- Q: 当多个节点对同一子任务给出冲突结果时如何取舍？ → A: 由裁判模型按质量评分取最高分结果，并记录全部候选来源与评分
- Q: 贡献点（奖励）的计量基准是什么？ → A: 贡献点 = 子任务市场价格 × 质量评分系数（以市场价为基础按质量调整）
- Q: 多个节点同时满足某子任务能力时如何选择？ → A: 综合评分匹配——能力满足前提下按市场价格与历史成功率加权择优
- Q: 任务完成的时延期望（SLA）是什么？ → A: 异步市场模型，无硬实时要求；在节点可用前提下 95% 任务 5 分钟内到达 Settled

## User Scenarios & Testing *(mandatory)*

本特性覆盖 IAI 系统的核心闭环：任务发起方提交任务，系统将其分解并匹配到合适的 AI
节点网络执行，聚合结果交付，并基于贡献完成结算与市场记账。以下用户旅程按优先级排序，
每个旅程均可独立交付价值与独立测试。

### User Story 1 - 任务发起方提交任务并获得聚合结果 (Priority: P1)

任务发起方描述一个需要 AI 能力的任务（如"分析这份报告并给出改进建议"），提交后无需关心
由哪些节点执行，系统自动完成解析、分解、匹配、执行与结果聚合，最终返回一份可用的聚合结果。

**Why this priority**: 这是 IAI 作为"AI 任务市场"对最终用户的核心价值。没有它，节点网络与
经济系统都失去服务对象。它本身即构成最小可用产品（MVP）。

**Independent Test**: 提交一个跨能力（如 reasoning + writing）的任务，验证系统返回一份
完整、可读、覆盖任务全部要求的聚合结果，且任务状态最终到达 Settled。

**Acceptance Scenarios**:

1. **Given** 任务发起方有一个明确的任务描述，**When** 提交任务，**Then** 系统生成任务并
   进入 Created 状态，返回可追踪的任务标识。
2. **Given** 一个已创建的任务，**When** 系统完成解析与分解，**Then** 任务被拆分为可由
   单一能力承担的子任务，并进入 Matched 状态。
3. **Given** 子任务已匹配到节点，**When** 节点完成执行，**Then** 系统聚合各子任务结果为
   一份统一交付物并标记 Aggregated。
4. **Given** 结果已聚合并通过质量验证，**When** 完成结算，**Then** 任务进入 Settled，
   发起方可查看完整结果与执行来源摘要。

---

### User Story 2 - 节点提供方注册能力并承接任务 (Priority: P2)

AI 能力提供方将其能力封装为标准化 IAI Node，声明能力与可用模型，加入网络后被自动发现并
匹配到相关子任务，完成执行后获得贡献点。

**Why this priority**: 没有节点供给，发起方的任务无处执行。它是供给侧的根基，但依赖
P1 定义的任务流才能体现价值，故为 P2。

**Independent Test**: 注册一个声明 `coding` 能力的节点，提交一个包含编码子任务的任务，
验证该子任务被匹配并分配给该节点，执行成功后该节点的贡献点增加。

**Acceptance Scenarios**:

1. **Given** 一个 AI 能力提供方，**When** 注册节点并声明 `node_id`、能力列表与模型列表，
   **Then** 节点进入可被发现的可用状态。
2. **Given** 网络中存在匹配某能力的可用节点，**When** 出现需要该能力的子任务，**Then**
   系统将子任务匹配给该节点。
3. **Given** 节点完成子任务执行，**When** 结果通过质量验证并结算，**Then** 该节点获得
   与其贡献对应的贡献点，并在账本中可追溯。

---

### User Story 3 - 结算可追溯与市场定价透明 (Priority: P3)

任务结算时，系统在账本中记录每次执行的贡献与奖励分发；定价基于公开的供需信号，参与各方
可查询价格与自身贡献记录。

**Why this priority**: 它保障激励体系的公平与信任，是网络长期健康的关键，但在 P1/P2
跑通前不产生独立价值，故为 P3。

**Independent Test**: 完成一个由多个节点协作的任务后，查询账本，验证每个参与节点的贡献
记录、奖励金额与任务执行记录一一对应，且价格依据公开规则可复算。

**Acceptance Scenarios**:

1. **Given** 一个已 Settled 的任务，**When** 查询账本，**Then** 可看到每个参与节点的
   贡献点与奖励分发记录，且与执行记录对应。
2. **Given** 市场中存在能力供需，**When** 任务进行定价，**Then** 价格依据公开的供需信号
   产生，且对参与各方对称可见。
3. **Given** 任意参与方，**When** 查询历史记录，**Then** 可获得自身全部贡献与结算明细，
   账本记录不可被篡改。

---

### Edge Cases

- 当任务描述无法被解析为任何已知能力时，系统应明确拒绝并告知发起方原因，任务不进入执行。
- 当没有任何可用节点匹配某子任务能力时，系统应将任务挂起或排队，并向发起方说明等待原因。
- 当某节点在执行中失败或超时，系统应重新匹配其他可用节点，单节点失败不导致整个任务失败。
- 当节点返回的结果未通过质量验证时，该结果不得进入聚合与结算，须重试或重新匹配。
- 当多个节点对同一子任务给出冲突结果时，系统 MUST 由裁判模型按质量评分取最高分结果，
  并记录全部候选来源及其评分以供审计。
- 当结算所需的执行证据缺失时，任务不得进入 Settled 状态。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 允许任务发起方以自然语言描述提交任务，并返回可追踪的任务标识。
- **FR-002**: 系统 MUST 将每个任务推进经过完整生命周期：Created → Parsed → Decomposed →
  Matched → Executed → Aggregated → Settled，且状态单向推进。
- **FR-003**: 系统 MUST 将任务解析并分解为可由单一能力承担的子任务。
- **FR-004**: 系统 MUST 依据子任务所需能力，将其匹配到声明了该能力的可用节点；当多个
  节点均满足能力要求时，MUST 采用综合评分匹配，在能力满足前提下按市场价格与历史成功率
  加权择优（不得使用纯随机或纯最低价单一策略）。
- **FR-005**: 系统 MUST 支持 AI 能力提供方注册标准化节点，声明唯一标识、能力列表与可用模型列表。
- **FR-006**: 系统 MUST 仅通过节点声明的能力契约进行匹配与交互，不依赖具体模型实现细节。
- **FR-007**: 系统 MUST 将各子任务的执行结果聚合为面向发起方的统一交付物。
- **FR-008**: 系统 MUST 在聚合进入结算前对结果执行自动质量门禁：先进行规则校验，再由
  裁判模型（model-as-judge）评分，达到可配置阈值方可通过；未通过者不得进入 Settled。
- **FR-009**: 系统 MUST 在单节点执行失败或超时时重新匹配，保证单点失败不导致任务整体失败。
- **FR-010**: 系统 MUST 为每次执行生成账本记录；无账本记录的执行不得进入 Settled。
- **FR-011**: 系统 MUST 依据贡献为参与节点计算并分发贡献点，计量基准为「子任务市场价格 ×
  质量评分系数」（以市场价为基础按质量评分调整），且贡献点与执行记录可追溯对应。
- **FR-012**: 系统 MUST 基于公开的供需信号对任务/能力定价，价格对参与各方对称可见。
- **FR-013**: 系统 MUST 保证账本记录不可篡改，并可供参与方独立核验。
- **FR-014**: 发起方 MUST 能够随时查询任务的当前状态与最终聚合结果及执行来源摘要。
- **FR-015**: 参与节点 MUST 能够查询自身的贡献与结算明细。
- **FR-016**: 系统 MUST 记录每次状态转移的可追溯信息（输入、负责节点、结果指纹），用于审计。

### Key Entities *(include if feature involves data)*

- **Task（任务）**: 发起方提交的工作单元。关键属性：任务标识、原始描述、当前生命周期状态、
  关联子任务集合、最终聚合结果引用。
- **Subtask（子任务）**: 任务分解后由单一能力承担的最小执行单元。关键属性：所需能力、
  分配节点、执行结果、质量验证状态。
- **IAI Node（节点）**: AI 能力容器。关键属性：唯一标识、能力列表、可用模型列表、可用状态、
  累计贡献点。
- **Ledger Entry（账本记录）**: 一次执行的不可篡改记账。关键属性：关联任务/子任务、参与节点、
  贡献点、奖励金额、执行证据指纹、时间戳。
- **Market Price（市场价格）**: 某能力在某时刻的定价信号。关键属性：能力类型、价格、供需依据、
  公示时间。
- **Result（结果）**: 子任务或聚合后的交付物。关键属性：内容、来源节点、质量评分、
  是否通过质量门禁（达阈值标记）、冲突候选来源及评分（如适用）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 任务发起方提交一个跨能力任务后，可在无需了解任何节点细节的情况下获得一份
  覆盖全部任务要求的聚合结果。
- **SC-002**: 至少 95% 的成功提交任务能完整走完生命周期并到达 Settled 状态。
- **SC-003**: 当任一参与节点在执行中失败时，至少 90% 的此类任务仍能通过重新匹配成功完成。
- **SC-004**: 100% 进入 Settled 的任务都具有与执行记录一一对应、可独立核验的账本记录。
- **SC-005**: 节点提供方完成一个子任务后，可在结算完成时查询到对应的贡献点增加。
- **SC-006**: 任意参与方可在公开规则下复算任一任务的定价结果，复算值与系统记录一致。
- **SC-007**: 未通过质量验证的结果在 100% 的情况下都不会进入结算。
- **SC-008**: 任务提交后立即返回任务标识；在节点供给充足前提下，至少 95% 的任务在 5 分钟内
  到达 Settled 状态（异步市场模型，无硬实时延迟要求）。

## Assumptions

- V1 范围聚焦"单任务从提交到结算"的核心闭环；跨任务的复杂工作流编排、节点声誉系统的
  高级算法不在本特性范围内。
- 结算以系统内部"贡献点（Contribution Credit）"计量，不涉及法币或链上代币兑换；外部
  价值兑换为后续特性。
- 任务发起方与节点提供方均为已认证的网络参与者；身份认证与准入由平台既有机制提供，
  不在本特性范围内。
- 质量验证机制已明确为「规则校验 + 裁判模型评分」的自动门禁（见 FR-008）；其通过阈值
  与裁判模型选型为可配置实现细节，本特性约束的是"未通过不得结算"这一行为契约。
- 网络中存在足够多样的节点供给以覆盖常见能力（reasoning、coding、writing 等）；供给
  不足时按 Edge Case 处理为挂起/排队。
- 价格的"公开供需信号"指对全体参与者对称可见的市场信息，具体定价公式为实现细节。
